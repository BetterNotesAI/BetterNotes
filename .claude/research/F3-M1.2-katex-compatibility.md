## 1. Contexto: qué genera GPT-4o para los 4 templates activos

Los 4 templates activos en BetterNotes usan los siguientes paquetes LaTeX en sus preambles:

| Template | Paquetes matemáticos |
|----------|---------------------|
| `2cols_portrait` | `amsmath`, `amssymb` |
| `landscape_3col_maths` | `amsmath`, `amsthm`, `empheq`, `amssymb`, `amsfonts`, `mathrsfs` |
| `study_form` | `amsmath`, `amssymb`, `amsfonts` |
| `lecture_notes` | `amsmath`, `amsthm`, `amssymb`, `amsfonts` |

Los style guides instruyen explícitamente a GPT-4o a usar:
- `align*`, `equation*` para display math
- `$...$` para inline math
- `\begin{definition}`, `\begin{theorem}`, `\begin{proposition}`, `\begin{obs}` (landscape_3col_maths)
- `\begin{workedexample}`, `\begin{keypoint}`, `\begin{warning}` (lecture_notes — tcolorbox)
- `\formulabox{Name}{formula}` — macro custom que usa `$#2$` internamente (study_form)
- `\nabla`, `\mathbf{}`, `\varepsilon`, `\mu_0`, `\partial` (ejemplos de electromagnetismo)
- `\det`, `\lambda`, `\mathbb{R}`, `\begin{pmatrix}` (ejemplos de álgebra lineal)
- `\frac{}{}`, `\lim`, `\sum`, `\int`, `\cdot`, `\Rightarrow` (cálculo)

### Macros custom definidas en preambles que KaTeX NO conoce
Estas son macros que el parser verá en el `.tex` pero que KaTeX no puede resolver por sí solo:

| Macro | Template | Definición | Impacto en KaTeX |
|-------|----------|------------|-----------------|
| `\dd` | landscape_3col_maths | `\mathrm{d}` | Fácil: expandir manualmente |
| `\real` | landscape_3col_maths | `\mathbb{R}` | Fácil: expandir manualmente |
| `\cplex` | landscape_3col_maths | `\mathbb{C}` | Fácil: expandir manualmente |
| `\HR` | 2cols_portrait, study_form | regla horizontal, no-math | No aplica (bloque de layout) |
| `\MyBox{...}` | 2cols_portrait | fbox con minipage | No aplica (bloque de layout) |
| `\sectionbar{...}` | study_form | colored rule | No aplica (bloque de layout) |
| `\formulabox{Name}{formula}` | study_form | colorbox + `$#2$` | La parte `$#2$` sí es math — solo extraer el argumento |

## 2. Análisis de compatibilidad KaTeX

### 2.1 Comandos que KaTeX soporta completamente

KaTeX soporta nativamente todos los comandos frecuentes en los templates:

**Operadores y funciones:**
`\frac`, `\sqrt`, `\sum`, `\int`, `\lim`, `\det`, `\max`, `\min`, `\log`, `\sin`, `\cos`, `\tan`, `\cdot`, `\times`, `\div`, `\partial`, `\nabla`

**Entornos de display math:**
`equation`, `equation*`, `align`, `align*`, `gather`, `gather*`
KaTeX soporta estos con `renderToString` + la opción `displayMode: true`.

**Fuentes matemáticas:**
`\mathbb{R}`, `\mathbb{C}`, `\mathbb{N}`, `\mathbb{Z}` (del paquete `amssymb`)
`\mathbf{}`, `\mathit{}`, `\mathrm{}`, `\mathcal{}`, `\mathsf{}`
`\mathfrak{}` (requiere KaTeX >= 0.11)

**Símbolos griegos:** todos los estándar: `\alpha`, `\beta`, `\gamma`, `\delta`, `\varepsilon`, `\lambda`, `\mu`, `\sigma`, `\omega`, `\Omega`, `\Sigma`, `\Pi` — soportados.

**Matrices y vectores:**
`\begin{pmatrix}`, `\begin{bmatrix}`, `\begin{vmatrix}`, `\begin{matrix}` — todos soportados.

**Flechas y relaciones:**
`\Rightarrow`, `\Leftrightarrow`, `\to`, `\mapsto`, `\leq`, `\geq`, `\neq`, `\subset`, `\in`, `\notin` — soportados.

**Decoradores:**
`\hat{}`, `\bar{}`, `\vec{}`, `\dot{}`, `\ddot{}`, `\tilde{}`, `\overline{}`, `\underline{}` — soportados.

### 2.2 Comandos problemáticos — lista negra relevante para BetterNotes

#### BLOQUEANTES REALES (comandos que GPT-4o puede generar, KaTeX no soporta)

| Comando | Frecuencia esperada | Estado KaTeX | Alternativa |
|---------|--------------------|----|-------------|
| `\mathrsfs` / `\mathscr{}` | Media (landscape_3col_maths importa `mathrsfs`) | **No soportado** | Usar `\mathcal{}` — GPT-4o raramente distingue |
| `\iint`, `\iiint`, `\oiint` | Media (integrales dobles/triples) | Soportado desde KaTeX 0.10+ | No hay problema |
| `\boldsymbol{}` | Alta (vectores en física) | **Parcialmente soportado** — funciona en modo math, falla fuera | Usar `\mathbf{}` como alternativa |
| `\bm{}` (del paquete `bm`) | Media | **No soportado** | Usar `\mathbf{}` |
| `\text{}` dentro de math | Alta | Soportado | Sin problema |
| `\operatorname{}` | Media | Soportado | Sin problema |
| `\underbrace{}`, `\overbrace{}` | Baja | Soportado | Sin problema |
| `\xleftarrow{}`, `\xrightarrow{}` | Baja | Soportado | Sin problema |

#### NO BLOQUEANTES — comandos de entorno que no son math

Estos comandos aparecen en los templates pero son estructurales de LaTeX, no fórmulas. El parser de bloques los separará correctamente y no se pasarán a KaTeX:

- `\begin{multicols*}`, `\begin{multicols}` — bloque de layout
- `\begin{tcolorbox}`, `\begin{workedexample}`, `\begin{keypoint}` — cajas de layout
- `\begin{theorem}`, `\begin{definition}`, `\begin{proposition}` — bloques de texto con contenido math dentro
- `\sectionbar{}`, `\formulabox{}`, `\MyBox{}` — macros de layout
- `\section*{}`, `\subsection*{}` — títulos

#### ENTORNOS AMSMATH que KaTeX sí soporta (importante confirmarlo)

| Entorno | Estado |
|---------|--------|
| `align*` | Soportado |
| `equation*` | Soportado |
| `gather*` | Soportado |
| `multline*` | Soportado |
| `cases` | Soportado |
| `split` | Soportado |
| `array` | Soportado |

#### ENTORNO `empheq` — el único paquete potencialmente problemático

`landscape_3col_maths` importa `\usepackage{empheq}` en su preamble. `empheq` sirve para enfatizar ecuaciones con cajas/corchetes. GPT-4o podría generar:

```latex
\begin{empheq}[box=\fbox]{align*}
  E = mc^2
\end{empheq}
```

KaTeX **no soporta `empheq`**. Sin embargo, revisando el `styleGuide` de ese template, GPT-4o es instruido a usar `align*` y `equation*` — no se menciona `empheq` en ningún ejemplo ni en el style guide. El riesgo es bajo pero existe.

## 3. Análisis por template

### 3.1 `2cols_portrait` — Compatibilidad: ALTA (95%)
- Math: inline `$...$` y `\MyBox{formula}` donde formula es math simple
- Comandos típicos: `\frac{}{}`, `\Delta`, `\frac{1}{2}mv^2`, fracciones, potencias
- Único riesgo: si GPT-4o usa `\boldsymbol{}` para vectores (poco probable, no está en el ejemplo)
- Veredicto: KaTeX viable sin ajustes

### 3.2 `landscape_3col_maths` — Compatibilidad: ALTA (90%)
- Math más densa: teoremas, proposiciones, `align*`, definiciones con límites, integrales
- Macros custom: `\dd`, `\real`, `\cplex` — requieren macros KaTeX (`macros` option)
- Paquete `mathrsfs`: si GPT-4o usa `\mathscr{}`, KaTeX falla. Riesgo moderado.
- Paquete `empheq`: presente en preamble pero no en style guide ni ejemplos. Riesgo bajo.
- Veredicto: KaTeX viable con 3 macros custom y prompt tweak para evitar `\mathscr`

### 3.3 `study_form` — Compatibilidad: MUY ALTA (97%)
- Math: `\formulabox{Name}{formula}` — el parser extrae el argumento `formula` y lo pasa a KaTeX
- Comandos típicos: `\nabla\cdot\mathbf{E}`, `\varepsilon_0`, `\mu_0` — todos soportados
- Veredicto: KaTeX viable sin ajustes

### 3.4 `lecture_notes` — Compatibilidad: ALTA (92%)
- Math: ecuaciones con `align*`, `equation`, matrices `pmatrix`, polinomios característicos
- El ejemplo muestra `\begin{pmatrix}`, `\det(A - \lambda I)`, `\mathbb{R}^{n\times n}` — todo soportado
- Entornos theorem/definition/remark: son texto con math inline — el parser los trata como bloques de texto
- Riesgo: `\boldsymbol{}` si GPT-4o lo usa para vectores en física (moderado)
- Veredicto: KaTeX viable con prompt tweak para vectores

## 4. Comandos NO soportados por KaTeX — lista negra completa relevante

| Comando/Paquete | Riesgo de aparición en BetterNotes | Estrategia recomendada |
|-----------------|-----------------------------------|------------------------|
| `\mathscr{}` (mathrsfs) | Moderado — solo landscape_3col_maths | Ajustar prompt: "use `\mathcal{}` not `\mathscr{}`" |
| `\bm{}` (paquete bm) | Bajo — no en ningún preamble | Ajustar prompt: "use `\mathbf{}` for bold vectors" |
| `\boldsymbol{}` | Moderado — física usa vectores | KaTeX 0.15+ lo soporta parcialmente; o prompt tweak |
| `\empheq` entorno | Bajo — preamble lo importa pero no el style guide | No usar en ejemplos |
| `\DeclareMathOperator` | Muy bajo — el preamble no lo incluye | Prompt: "use `\operatorname{}`" |
| `\usepackage{physics}` | Muy bajo — no en preamble | No cambia nada |
| `\dv{}{}`, `\pdv{}{}` | Muy bajo — paquete physics | No aplica |

## 5. Estrategia concreta para casos no soportados

### Opción A (RECOMENDADA): Macros KaTeX + prompt adjustment

**Paso 1 — Macros KaTeX para las custom del preamble:**
```js
const KATEX_MACROS = {
  "\\dd": "\\mathrm{d}",
  "\\real": "\\mathbb{R}",
  "\\cplex": "\\mathbb{C}",
};
// Pasar a katex.renderToString(formula, { macros: KATEX_MACROS })
```

**Paso 2 — Añadir una línea al system prompt de GPT-4o:**
```
- MATH RENDERING: Use \mathbf{} for bold vectors (not \boldsymbol or \bm).
  Use \mathcal{} for script letters (not \mathscr).
  Use standard align*/equation* environments (not empheq).
```

Este ajuste es mínimo, no rompe la estructura del prompt existente, y elimina el 95% del riesgo.

### Opción B: Fallback silencioso

Si KaTeX lanza un error al renderizar una fórmula, capturar el error y mostrar el LaTeX crudo en monospace como fallback visual. El usuario ve la fórmula, aunque sin renderizado bonito.

```jsx
try {
  return katex.renderToString(formula, { throwOnError: true, macros });
} catch {
  return `<code class="katex-fallback">${formula}</code>`;
}
```

### Opción C: MathJax como alternativa

MathJax soporta prácticamente todo pdflatex incluyendo `empheq`, `mathrsfs`, `bm`, etc. Sin embargo:
- Es ~3x más lento que KaTeX en renderizado inicial
- Bundle size ~400KB vs ~280KB KaTeX
- API async vs sync de KaTeX
- Integración con React más compleja

**Conclusión sobre MathJax**: no justificado para BetterNotes dado el nivel de control que tenemos sobre el output de GPT-4o. Si surgieran bloqueantes reales post-implementación, es la opción de escalado.

## 6. Conclusión — ¿KaTeX es viable?

**SÍ. KaTeX es completamente viable como motor de rendering para BetterNotes.**

Razonamiento:
1. El 90-97% de los comandos matemáticos que GPT-4o genera con los 4 templates están soportados nativamente por KaTeX.
2. Los 3 macros custom (`\dd`, `\real`, `\cplex`) se resuelven con la opción `macros` de KaTeX — 3 líneas de código.
3. Los únicos comandos problemáticos (`\mathscr`, `\boldsymbol`, `\bm`) se eliminan con una adición mínima al system prompt de GPT-4o.
4. El paquete `empheq` aparece en un preamble pero no en ningún ejemplo ni style guide — GPT-4o no lo usará de forma espontánea.
5. No hay ningún bloqueante real que requiera MathJax.

**Acciones necesarias para F3-M1.2 → F3-M1.3:**
1. Añadir al system prompt: instrucción de usar `\mathbf{}` y `\mathcal{}` en vez de `\boldsymbol`/`\mathscr`
2. Definir objeto `KATEX_MACROS` con 3 entradas para las macros custom
3. Implementar fallback `throwOnError: false` o catch visual para cualquier edge case residual
4. No se requiere MathJax ni cambio de motor
